2. ARSITEKTUR TINGKAT TINGGI
======================================================================

Gunakan arsitektur modular. Untuk tugas besar, monorepo modular lebih mudah dibanding microservices penuh. Namun boundary domain harus jelas.

Opsi arsitektur monorepo:
/apps
  /api-gateway
  /centralbank-core
  /wallet-api
  /marketplace
  /pos
  /supplierhub
  /logistikita
  /umkm-insight
/packages
  /common
  /auth
  /ledger
  /money
  /testing

Opsi sederhana dalam satu backend modular:
/src/modules
  auth/
  centralbank/
  wallet/
  ledger/
  payments/
  loans/
  fees/
  marketplace/
  pos/
  supplierhub/
  logistics/
  analytics/
  gateway/
  audit/

Komponen utama:
1. API Gateway/Integrator
   - Validasi JWT.
   - Routing request antar modul/service.
   - Request logging.
   - Rate limiting opsional.
   - Idempotency-key enforcement untuk transaksi finansial.

2. CentralBank Core
   - Issuance CBDC.
   - Burn/redemption CBDC.
   - Reserve account.
   - Wallet account master balance.
   - Double-entry ledger.
   - Payment settlement.
   - Transfer user-to-user.
   - Fee/tax calculation.
   - Loan issuance dan repayment.
   - Monetary policy event: stimulus, distribution, supply audit.

3. SmartBank Wallet
   - Registrasi dan login user.
   - KYC profile sederhana.
   - Dashboard saldo.
   - Riwayat transaksi.
   - Transfer antar user.
   - Bayar invoice/payment request.
   - Ajukan pinjaman.
   - Notifikasi status pembayaran.
   - UI/UX boleh terinspirasi dari pola mobile banking: home saldo, mutasi, transfer, QR/payment, inbox, profile/security.

4. Marketplace
   - Seller CRUD produk.
   - Buyer browse produk.
   - Cart/checkout.
   - Membuat order.
   - Mengirim payment request ke CentralBank Core.
   - Setelah settled, trigger LogistiKita.

5. POS
   - Kasir input item.
   - Generate tagihan/invoice.
   - Pembayaran via wallet/CentralBank Core.
   - Simpan histori transaksi POS.

6. SupplierHub
   - Kelola stok bahan baku.
   - UMKM order bahan.
   - Supplier konfirmasi stok dan harga.
   - Payment request ke CentralBank Core.
   - Opsional trigger pengiriman bahan via LogistiKita.

7. LogistiKita
   - Request pengiriman.
   - Hitung ongkir.
   - Payment request ongkir.
   - Tracking status pengiriman.

8. UMKM Insight
   - Read-only dari ledger CentralBank Core.
   - Dashboard penjualan.
   - Cashflow.
   - Performa merchant/UMKM.
   - Subscription premium 10.000/minggu.
   - Tidak boleh mengubah saldo atau transaksi.

9. API CONTRACT REKOMENDASI
======================================================================

Semua response JSON gunakan format konsisten:
{
  "success": true,
  "data": {},
  "error": null,
  "meta": {
    "request_id": "...",
    "timestamp": "..."
  }
}

Error response:
{
  "success": false,
  "data": null,
  "error": {
    "code": "INSUFFICIENT_BALANCE",
    "message": "Saldo tidak mencukupi",
    "details": {}
  },
  "meta": {
    "request_id": "...",
    "timestamp": "..."
  }
}

Headers finansial wajib:
- Authorization: Bearer <jwt>
- X-Request-Id: unique id per request
- Idempotency-Key: wajib untuk POST transaksi
- X-Service-Name: untuk service internal

9.3 Transfer

POST /api/v1/transfers
Request:
{
  "to_wallet_id": "wal_receiver",
  "amount": 10000,
  "note": "bayar makan"
}
Process:
- Validate JWT.
- Validate amount > 0.
- Check cooldown and max daily transaction.
- Quote bank fee/gateway fee/tax if applicable.
- Check payer balance.
- Write ledger atomically.
Response:
{
  "transaction_id": "trx_...",
  "status": "SETTLED",
  "amount": 10000,
  "fees": {
    "bank_fee": 100,
    "gateway_fee": 50,
    "tax": 200
  }
}

9.4 Payment Request

POST /api/v1/payment-requests
Request from app/service:
{

  "gross_amount": 100000,
  "description": "Checkout order ord_123",
  "metadata": {
    "order_id": "ord_123"
  },
  "expires_at": "2026-12-31T10:00:00Z"
}
Response:
{
  "payment_request_id": "payreq_...",
  "status": "PENDING",
  "amount_due": 105500,
  "fee_breakdown": {}
}

POST /api/v1/payment-requests/{id}/pay
Request:
{
  "wallet_pin": "123456"
}
Response:
{
  "transaction_id": "trx_...",
  "status": "SETTLED"
}

9.6 Loan

POST /api/v1/loans/apply
Request:
{
  "amount": 100000,
  "purpose": "modal UMKM"

Validation:
- amount <= 100000
- outstanding loan user + amount <= 100000
- reserve cukup

  "loan_id": "loan_...",
  "principal": 100000,
  "interest": 10000,
  "total_due": 110000,

Request:
{
  "amount": 110000
}
Response:
{
  "status": "PAID",
  "remaining_due": 0

9.7 Marketplace Checkout

POST /api/v1/marketplace/checkout
Request:
{
  "buyer_wallet_id": "wal_buyer",
  "items": [
    {"product_id": "prd_1", "qty": 2}
  ],
  "shipping_address": "..."

Process:

- Calculate subtotal.
- Create order PENDING_PAYMENT.

12. PRIVACY, SECURITY, DAN COMPLIANCE DESIGN
======================================================================

Authentication:
- Password wajib di-hash dengan bcrypt/argon2.
- JWT access token pendek, refresh token lebih panjang.
- Untuk transaksi wallet, gunakan PIN atau second factor sederhana.

Authorization:
- Role-based access control.
- Object-level authorization untuk wallet, order, shipment, supplier order.

Idempotency:
- Semua POST transaksi harus menerima Idempotency-Key.
- Idempotency-Key unik per user/service + endpoint + payload hash.

Rate limit:
- Login endpoint rate-limited.
- Payment endpoint rate-limited.
- Gateway bisa menerapkan cooldown 10-30 detik per user untuk transaksi finansial.

Audit:
- Log semua request finansial.
- Log semua admin action.
- Jangan log password/PIN/token penuh.

Data privacy:
- Merchant melihat status pembayaran, bukan seluruh identitas finansial pembeli.
- Analytics memakai agregasi.
- Admin audit detail transaksi harus memakai reason_code.

CBDC safeguards:
- Jangan membuat fitur freeze arbitrer tanpa proses audit. Jika diperlukan untuk prototype, beri status SUSPENDED pada wallet dengan reason dan audit log, bukan menghapus saldo.
- Jangan membuat aturan uang yang membuat unit CBDC berbeda-beda per orang. Kontrol dilakukan di level payment/request/policy, bukan di token uang.
- Pastikan semua uang tetap fungible.

======================================================================
13. NON-FUNCTIONAL REQUIREMENTS
======================================================================

Correctness:
- Tidak ada transaksi settled tanpa ledger entries valid.
- Tidak ada ledger imbalance.
- Tidak ada saldo negatif tanpa account teknis yang diizinkan.
- Total supply invariant harus bisa diaudit.

Performance:
- Untuk prototype, target transaksi sederhana < 500 ms lokal.
- Gunakan index pada wallet_id, transaction_id, created_at, source_app.

Reliability:
- Gunakan database transaction.
- Gunakan retry aman hanya dengan idempotency.
- Jangan retry settlement tanpa idempotency key.

Maintainability:
- Pisahkan controller, service, repository, domain model.
- Jangan taruh business logic di controller.
- Buat unit test untuk fee, settlement, loan, supply invariant.

Observability:
- request_id di setiap log.
- audit_logs untuk business events.
- error code konsisten.

======================================================================
15. ACCEPTANCE CRITERIA
======================================================================

Minimum viable system dianggap selesai jika:
1. User bisa register/login dan mendapat wallet.
2. Wallet awal mendapat 50.000 selama distribusi awal tidak melebihi 2% supply.
3. Central reserve tetap >= 98% pada fase awal.
4. User bisa melihat saldo dan mutasi.
5. User bisa transfer ke user lain.
6. Marketplace bisa membuat product, checkout, payment request, dan settlement.
7. POS bisa generate invoice dan menerima pembayaran.
8. SupplierHub bisa membuat order bahan dan payment request.
9. LogistiKita bisa membuat shipment dan biaya pengiriman.
10. UMKM Insight bisa membaca ledger dan menampilkan summary.
11. Semua transaksi settled punya ledger double-entry seimbang.
12. Fee bank, gateway, marketplace/POS/supplier/logistik, dan pajak dihitung sesuai aturan.
13. Loan maksimal 100.000/user dan bunga 10%.
14. Cooldown dan max 10 transaksi harian diterapkan minimal pada endpoint transfer/payment.
15. Tidak ada service selain CentralBank Core yang bisa mengubah saldo.

16. TEST CASE WAJIB
======================================================================

16.1 Auth & wallet
- Register user baru -> wallet dibuat -> initial balance 50.000 jika supply policy mengizinkan.
- Login berhasil -> JWT valid.
- Password salah -> error UNAUTHORIZED.

16.2 Ledger
- Transfer 10.000 menghasilkan debit payer dan credit receiver + fee accounts.
- Total debit sama dengan total credit.
- Ledger tidak bisa dihapus/update melalui API.

16.3 Supply
- Total money supply tidak melebihi 1.000.000.000.
- Distribusi awal tidak melewati 2% supply.
- Reserve awal minimal 98%.

16.4 Payment request
- Marketplace checkout menciptakan payment request PENDING.
- Pay payment request dengan saldo cukup -> SETTLED dan order PAID.
- Pay payment request expired -> PAYMENT_EXPIRED.
- Idempotency retry dengan key sama -> response transaksi sama, bukan transaksi baru.

16.5 Fee
- Marketplace amount 100.000 -> marketplace fee 2.000.
- Bank fee 1% -> 1.000.
- Gateway fee 0,5% -> 500.
- Pajak 2% -> 2.000.
- Gunakan pembulatan konsisten floor.

16.6 Loan
- Apply loan 100.000 -> approved jika belum punya outstanding loan dan reserve cukup.
- Apply loan 101.000 -> LOAN_LIMIT_EXCEEDED.
- Repay loan 110.000 -> status PAID.

16.7 Limits
- User transaksi > 10 kali/hari -> DAILY_LIMIT_EXCEEDED.
- User transaksi dalam cooldown -> COOLDOWN_ACTIVE.

16.8 Analytics
- UMKM Insight GET dashboard tidak mengubah saldo atau ledger.
- Analytics hanya menghitung transaksi SETTLED.
